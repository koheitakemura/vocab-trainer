## TL;DR

- **1つの語彙学習エンジン × 複数コース**。学習ロジック（出題・復習・進捗）は全コース共通で、違いは語彙データと表示言語だけ
- **利用者は複数だが各自が個人用**：チームリーダーボード等の社会的機能は作らない。各人が自分の進捗だけを持つ
- 2系統5コース：
  - **Kohei 系**: 英語 10k→30k（配置テスト＋マイニング型）／タガログ語 0→3k（レール型）
  - **メンバー系**: 日本語 0→3k（出荷済み）・3k→10k・10k→30k（レール型→cloze→較正、JLPTバッジ付き）
  - コースC（日本語0→3k）出荷後は**直列**で1コースずつ着手（Phase2→3→4→5。並行案は2026-07-15に撤回。理由は §7・判断ログ#14）
- 復習エンジン = **FSRS-6**（Anki 採用の最新式。MIT の ts-fsrs で合法組込）。SRS は自作しない
- **Teams 連携 = 単語収穫**：チャットでやりとりした／分からなかった単語をアプリに取り込んで学習カード化。通知・順位表は作らない
- **サーバーレス**: バックエンド無し。Vite 静的 SPA を個人 GitHub → GitHub Pages で無料公開。進捗は端末内 IndexedDB＋手動 JSON バックアップ（クラウド同期なし）
- データは**3言語とも再配布可能なオープンデータのみで構築**（ライセンス一次ソース検証済み）。GitHub Pages 公開＝再配布のため研究/教育限定データ（BCCWJ・jpdb）は不使用

## 2. アーキテクチャ方針: 1エンジン × 複数コース

**判断基準**: 競合10製品（Anki/jpdb/WaniKani/iKnow!/Renshuu/Clozemaster/Memrise/LingQ/Migaku/WordUp）調査で、成功例は全て次の4点に収斂。

1. **単語知識は「デッキ」でなく「ユーザー」に付く**（最重要のアーキ判断。jpdb の10状態モデルが参考: New / Learning / Known / Never-Forget / Blacklisted / Suspended / Redundant）→ 重複出題排除・カバレッジ統計・コース間移動が全部ついてくる
2. **頻度順の提示**（語彙学習 = 「有用度順の有限集合の欠損を埋める問題」）
3. **文脈（例文・クローズ）での定着**（孤立単語カードだけにしない）
4. **近代SRS**（FSRS。もはや差別化要素でなく前提）

## 3. コア機能（全コース共通エンジン）

### 3.1 復習エンジン（FSRS-6）
- ライブラリ: **ts-fsrs**（MIT・v5.4.1/2026-05・pure TS でブラウザ動作）。SM-2 比で同一保持率を約20-30%少ないレビューで達成
- ユーザー設定は **desired retention 1個だけ**（既定0.9、上級は0.85推奨表示）
- 評価 = **Again/Good の2択**既定（4択オプション）。初見「もう知ってる」→ 長期間隔
- leech 検出（lapse≥8）で自動サスペンド提案
- 約1,000レビュー後にパラメータ個人最適化（fsrs-browser WASM・2.4万レビュー≈3.5秒でクライアント側完結）

### 3.2 配置テスト（10k→30k コースの生死を決める）※コースA・E
- **Yes/No テスト＋擬似語補正**（学術定番。補正式 = (h−f)/(1−f)）
- 頻度1k帯ごとに実在語サンプリング（計60-120語・3-5分）→ **既知フロンティア（rank N 以下を一括「既知」化）**決定 → 既知1万語の学習者に既知語を挽かせない
- 第2段: フロンティア直下で4択意味マッチ10-20問→境界±1帯調整。以後 FSRS が自己修正
- **日本語注意**: 漢字複合語は字面から意味推測でき Yes/No 誤検出増（実証あり）→ 意味マッチ比重を上げる
- 0開始コース（B・C）は配置テスト不要 = レール MVP が軽い

### 3.3 単語状態ストア＋カバレッジ可視化
- ユーザー単位 word-state DB（**IndexedDB・ローカルファースト**）。サーバー同期は行わず、**手動 JSON エクスポート/インポート**でバックアップ・端末移行（「進捗を書き出す/読み込む」ボタン）
- 常設メーター: 「推定語彙数 12,840語」（単一数値の成長メーター）＋「あと1,000語で一般記事の理解が x%→y%」（jpdb/WordUp式 payoff）
- 「卒業（Burned）」状態で既知確定語をレビューキューから除去（3万語規模ではキュー管理が生死を分ける）

### 3.4 カードの中身（全語必須の品質バー）
- 見出し語＋グロス＋**例文2-3個**＋**発音表記**＋品詞。iKnow! Core の実証済みバーから音声を発音表記に置換
- 例文3段構え: **Tatoeba（最クリーン）→ JESC（字幕由来280万英日ペア・CC BY-SA）→ LLM生成（「AI生成」フラグ付き）**
- **音声は不採用（2026-07-13 判断）**。事前生成TTSは1万→3万語帯だけで容量試算 約0.9-1.9GB となり GitHub Pages の1GB上限を単独で超過（§6 参照）。無料の Web Speech API 版も含め音声機能そのものを撤去し、**発音表記**（後述）に一本化。データ量ゼロ・追加ホスティング不要で解決する
- **発音表記**（見出し語の下に常設表示。`VocabCard.reading` フィールドを流用）:
  - 日本語 = かな読み（JMdict由来。実装済み・追加作業不要）
  - 英語 = IPA発音記号（**CMUdict**由来。ARPABET→IPA変換テーブルを1回作れば静的データとして無料生成可。§5.1 で既に参照元として記載済み）
  - タガログ語 = アクセント位置付き表記（áso/asó 等。§4.1 で元々計画していたアクセント表示と統合）

### 3.5 Teams 連携 = 単語収穫パイプライン
**目的**: ①Teams チャットでやりとりした単語をアプリに反映 ②分からない単語をすぐ調べて学習キューへ。通知・順位表は作らない。両系統（Kohei の英語・メンバーの日本語）で同じ導線が効く。

1. **MVP: 貼り付けマイニング（統合ゼロ・当日から使える）**
   - Teams でコピー→アプリの取り込みボックスに貼付→言語自動判定→未知語抽出（既知・学習中は自動除外）→意味・例文・発音表記付きプレビュー→ワンタップでカード化 or 辞書引きだけ
   - **辞書引き（JMdict/EJDict 同梱）は完全オフラインで動く**。同梱辞書に無い新語の意味/例文のその場生成だけは外部呼び出しが要る → MVP は「同梱辞書に有る語は即カード化」、新語の LLM 生成は各自の API キーをブラウザに入れる方式で後付け（サーバーは持たない）
   - §4 の「個人マイニング」機能そのもの。URL・PDF・任意テキストにも同じ導線
2. **Phase 2: クイック辞書引き**
   - アプリ内オムニ検索（打つ/貼ると語義・頻度ランク・自分の学習状態を即表示→「学習キューへ」）。**PWA 化**でタスクバー常駐＝Teams の隣に置く
3. **Phase 3（任意）: Graph API 自動取り込み**
   - 各自の delegated 認証（Chat.Read・個人サインイン＝admin consent 不要方向）で自分のチャットを日次取得→未知語候補を朝のレビュー前に提案。**MSAL.js の PKCE でクライアントのみ完結＝バックエンド不要**なのでサーバーレスのまま追加可
   - 貼り付けで不満が出てから作る。参考: 旧 incoming webhook は 2026-05 廃止、通知が欲しくなれば Workflows webhook（無料・15分）で追加可

### 3.6 進捗・習慣化
- ダッシュボード: 推定語彙数の成長曲線・ストリーク・日次レビュー完遂
- ゲーミフィケーションは **SRS の外側**（間隔の科学を汚染しない・競合共通則）

## 4. 帯域別の学習体験（0→3k と 10k→30k は別プロダクト級に違う）

### 4.1 レール型: 0→3k（コース B タガログ語・C 日本語）
- **選択肢ゼロ**: 「開始」を押すだけ（iKnow!式）。ゲート進行（WaniKani式）で順序強制。自由度は完走率の敵
- **reading-first**（2026-07-13 音声不採用に伴い変更）: 発音表記→意味が既定の初回提示。絵/写真マッチ（具象語）・ミニマルペア聞き分け
- 受容→産出: L2→意味カードが安定（stability>7日）で 意味→L2 産出カードを自動追加
- 例文は「表示するが問わない」添え物（初期は単語単位カードが最効率という実証研究）
- **日本語コース(C)固有**: ①かなブートキャンプ（先行モジュール・ローマ字は2-4週で強制フェードアウト）②**語彙駆動の漢字導入**（jpdb式: 学ぶ語に必要な字だけ動的に。データ: KANJIDIC2＋KRADFILE＋KanjiVG筆順・全てオープン）③ふりがな表示制御
- **タガログ語コース(B)固有**: 接辞・語根表示（mag-/-um-/-in 型。同語根の派生語を「家族」で提示）、アクセント位置表記（áso/asó で意味変化。§3.4 の発音表記で対応）
- ワークロード: 新規9-10語/日 → レビュー60-80枚/日 → **1日20-30分で1年=3,000語**。最初の2週は新規5語/日ランプアップ。1日2セッション（朝レビュー/夜新語）推奨
- 区切り: 0-1k（会話の約80-85%被覆）/ 1k-3k（約95%）をマイルストーン演出

### 4.2 cloze 主軸: 3k→10k（コース D 日本語）
- 単語カードで習得 → stability 閾値超えで**文脈クローズへ自動昇格**（カード総数は増やさない）
- コロケーション導入・多義語の語義弁別・既知語率可視化リーディング（未知語タップ→デッキ追加＝マイニング入口）
- 新規10-20語/日・DR 0.85 既定・1日20-40分。7,000語=15語/日で約1.3年
- 到達目標明示: 「Band 2 完了 = 字幕なし視聴・一般書講読の入口」（8-9k語で書き言葉98%被覆）

### 4.3 較正＋マイニング型: 10k→30k（コース A 英語・E 日本語）
- **初回体験 = 配置テストで既知語一括スキップ**（WordUp の Knowledge Map 較正が直接の先行例）
- 出題形式: コロケーション cloze（"hurl accusations"型）・類義語ニュアンス4択（mitigate/alleviate/curb…）・語族展開（語根→派生形）・モノリンガル定義切替
- **個人マイニング**（この帯の新語供給の主経路）: Teams チャット・業務文書・記事の貼付→未知語抽出→ワンタップ・カード化（§3.5）。頻度リスト供給は10-15kで打ち切り、以降マイニング＋職能別パック（HR/ビジネス）へ（汎用頻度リストは15k超で信頼性崩壊）
- **現実補正**: 10k→30k の2万語は15語/日で約4-5年。20k語族≈一般書ほぼ精読可、15-20k超の限界効用は「語数」より **depth（コロケーション正答率）**へ → プログレスバーは語数＋depth の2本立て
- 新規10-15語/日＋DR 0.80-0.85 既定。忙週は新規0でレビューのみ（FSRS は中断後の再スケジュールに強い）

## 5. データパイプライン（ライセンス一次ソース検証済み・社内/個人利用前提）

### 5.1 英語（コースA: 10k→30k・グロス=日本語）
| 層 | 採用 | 備考 |
|---|---|---|
| 頻度 | wordfreq en large（31.9万語形・rank40kまで実測確認） | **レンマ化必須**（spaCy/lemminflect→固有名詞除去→辞書照合）。2021年凍結だが上級語彙では実害小 |
| 既知語底面 | NGSL＋NAWL＋CEFR-J A1-B2（計約8千語）を既知として底面カット | 実効2万語前後に絞る |
| グロス | EJDict-hand（4.56万見出し・CC0）→日本語WordNet→kaikki→LLM生成の4段 | 品質確認は複数AIによる相互検証（人手ネイティブレビュー無し。判断ログ#18） |
| 例文 | Tatoeba（英日11.7万ペア）→ JESC（280万）→ LLM生成 | 15k-30k帯の過半は Tatoeba に例文なし→JESC＋LLM 必須 |
| 発音表記 | **CMUdict**（ARPABET）→ IPA 変換テーブル経由で表示 | 音声は不採用（2026-07-13）。データ量ゼロ・無料 |

### 5.2 タガログ語（コースB: 0→3k・グロス=日本語/英語）
| 層 | 採用 | 備考 |
|---|---|---|

## 6. 技術スタック（サーバーレス・100% クライアントサイド）

- **フロント**: **Vite＋React＋TypeScript の静的 SPA**（バックエンド無し。Next.js のサーバー機能は使わないため純粋な静的ビルドの Vite を採用）。**PWA 化**して「Teams の隣に常駐する辞書＋学習アプリ」に
- **配布/ホスティング**: **個人 GitHub の public リポジトリ → GitHub Pages（無料）**。メンバーは URL を開くだけ、PWA でインストールも可。アプリサーバー・DB は一切なし。※GitHub Pages の公開サイトは常に「URL を知れば誰でも閲覧可」（閲覧制限は Enterprise 専用）＝§5.4 の再配布クリーンが前提
- **進捗データ**: IndexedDB（Dexie）**ローカルファースト＝各人の進捗はその端末に**。アカウント無し。クラウド同期はせず、**手動 JSON エクスポート/インポート**でバックアップ・端末移行（データ消失対策）
- **語彙データ**: 静的 JSON パック（コース別・1k帯別に分割・遅延読込）。**音声は不採用**（2026-07-13）＝発音は `reading` フィールドの発音表記（かな/IPA/アクセント付き表記）で伝える
- **容量試算（音声不採用の根拠）**: 1万→3万語帯（2万語）に事前生成TTSを載せた場合、48kbps mono 換算で1語あたり約70KB（見出し語+例文2-3個）×2万語 ≈ **1.33GB** となり GitHub Pages の1GB上限を単独で超過（低音質32kbpsでも約0.94GBでほぼ余地なし）。発音表記に切り替えたことでこの制約自体が解消し、別ホスティングの検討は不要になった
- **SRS**: ts-fsrs（MIT）＋ fsrs-browser（WASM）。すべてブラウザ内で完結（サーバー不要）
- **データパイプライン**: Python ローカルバッチ（wordfreq / Sudachi / spaCy / 頻度集計 / CMUdict変換等）を**手元で実行**し、成果物（静的 JSON）だけをリポジトリに置く。実行時サーバーは無い
- **Mock-First**: UIトラック（モックデータ50語で学習ループUIを磨く）とデータspikeトラック（パイプライン検証）を並走→合流

## 9. 主要リスクと対策

| リスク | 対策 |
|---|---|
| 10k-30k 帯の例文カバレッジの崖（Tatoeba では過半ゼロ） | JESC 第2源＋LLM 生成（AI生成フラグ）の3段構え |
| wordfreq 頻度/レンマ化品質（rank20k超はノイズ増。日本語も BCCWJ→wordfreq/OPUS に変更） | Phase 0 で実測してから帯上限を確定 |
| タガログ語の 2k→3k 域はデータが薄い | McFarland 骨格＋コーパス統合。3kに届かなければ無理に水増しせず届いた語数で出す（判断ログ#19） |
| メンバー配布時の学習継続 | ワークロード数字を最初に見せる＋ストリーク＋JLPTバッジ＋Teams マイニングで業務直結 |
| 貼り付けマイニングが面倒で使われない | Phase 6 のクイック辞書引き（PWA常駐）で摩擦低減、それでも不満なら Graph API 自動化 |
| 進捗データの消失（ローカルのみ・ブラウザ消去で消える） | 手動 JSON エクスポート/インポート導線を目立たせ、初回オンボーディングで案内。サーバー同期はしない方針 |
| public 公開で再配布不可データを漏らす | データ層を最初から再配布可能のみに限定（BCCWJ・jpdb 不使用）。public リポジトリにシークレットを置かない |
| **コンテンツ品質＝人手ネイティブレビュー無し（判断ログ#18）** | タガログ語グロス・英語グロス・例文・コーチ文の品質上限はAI自身の多言語能力に依存し、人手検証では捕まる誤りが残る可能性がある。生成→別AIによる敵対的検証（相互検証）を必須プロセスとし、明らかな誤生成の検出だけでも機械的に行う。品質問題が使用中に見つかった場合の訂正導線（フィードバック）を用意する |
| **カテゴリ分類・コーチ文に自動生成スクリプトが存在しない** | コースCの208文・28カテゴリはAI著者セッションの一回限りの成果物で、生成コストは**コースごとに毎回まるごと発生**する（自動化されていない）。各コースのタスクリスト（2.5/3.5/4.5/5.5）に明示のLサイズ工数として計上済み。過小評価しない |
| **共通基盤（Phase 1.5）を後回しにした場合の手戻り** | コース切替UI・引用元表示・カバレッジ計算・i18nを2コース目以降に個別対応すると、後から3コース分やり直す事態になり得る。判断ログ#15によりPhase 1.5として先に一括対応する方針を採用済み |
| ja-0-3k のグロス言語がパイプラインスクリプト・PLAN.mdと食い違ったまま | 判断ログ#16でタガログ語を正と確定。Phase 3.1 で `build_ja_0_3k.py` 系のスクリプト自体にタガログ語訳ステップを正式に組み込み、以後の手翻訳（一回限りの手作業）を無くす |
| PWAが全コースのJSONを全訪問者に配信（vite.config.ts の既知の未解決課題） | Phase 1.5.8 でグロブ全指定からruntimeCachingへ移行。5コース分になった時点で影響が実害化するため後回しにしない |

## 10. 差別化メモ（将来の外販オプション・参考）

- タガログ語×頻度順SRSの本格アプリは実質空白市場（本格派は Clozemaster/LingQ のみ・日本語UIはほぼ皆無）。コースB は将来外販候補。**データ層は最初から再配布クリーン（BCCWJ・jpdb 不使用）なので、外販時もデータ差し替えが不要**（独自例文の権利だけ確認すれば商用移行できる）
- メンバー用日本語コース（JLPTバッジ付き）は比国の就労日本語需要に直結＝社内 HR 施策としても価値。Accenture 内での横展開余地
