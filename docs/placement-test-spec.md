## 0. 前提の確認

- コースA・Eの語彙データは 1k帯ごとの静的JSON（`words-10000-11000.json` … `words-29000-30000.json`、帯インデックス **10〜29の20帯**）としてビルド時に生成済み・配布済みという前提（`pipeline/emit.py` の `band_filename` 形式）。`frequencyRank` は絶対ランク（1始まりの言語全体順位）。
- `realRepository.getCards(courseId)` は該当コースの**全帯を一括fetch**して返す（`manifest.json` の bands を Promise.all で並列取得・結合）。つまり配置テストが始まる時点で、対象コースの全カード（10k〜30k分、最大2万語）は**既にメモリ上にある**——実在語のアイテムバンクは新規データ生成もfetchも不要で、ロード済み配列から実行時にサンプリングするだけで足りる（§2.1）。
- 新規に生成が必要なのは**擬似語（pseudoword）のみ**（§2.2）。これはビルド時にPythonパイプラインで1回生成し、小さな静的JSONとしてコースごとに配布する（§2.3）。

## 1. Yes/Noテスト＋擬似語補正（学術的定番手法）

### 1.1 出典と方式

Meara & Buxton (1987) が提案した Yes/No 語彙テスト（後継: Eurocentres Vocabulary Size Test / X_Lex・Y_Lex）。受験者に単語リストを見せ、各語について「知っている（Yes）／知らない（No）」を意味の説明なしで即答させる——4択や翻訳を要求しないため出題コストが極めて低い（1語1〜2秒）。

弱点は**自己申告バイアス**：知ったかぶりで Yes を連発する受験者がいると、素点（Yesと答えた率）だけでは実際の語彙力を過大評価する。これを検出するため、リストに**擬似語（pseudoword＝実在しないがその言語らしい形の単語）**を紛れ込ませる。擬似語に Yes と答える率が高い受験者ほど、実在語への Yes 率も「本当に知っている」以外の要因（当てずっぽう・過大申告）で膨らんでいると推定できる。

### 1.2 補正式の意味

```
p = (h − f) / (1 − f)
```

- **h（hit rate）**: 実在語に正しく Yes と答えた率（0〜1）
- **f（false-alarm rate）**: 擬似語に誤って Yes と答えた率（0〜1）
- **p（corrected score）**: 補正後の「本当に知っている」推定率

**導出の直感（high-threshold model）**: 受験者の反応は2種類に分解できると仮定する——①本当にその語を知っていて確信を持って Yes と答える（確率 p）、②知らない語には「知っている風の見た目」に釣られて確率 f で当てずっぽう Yes と答える。実在語を見たとき Yes と答える確率は
```
h = p + (1 − p) × f
```
（「本当に知っている（p）」+「知らないが当てずっぽうでYesと言う（(1−p)×f）」）。これを p について解くと
```
p = (h − f) / (1 − f)
```
になる。つまりこの式は「観測されたYes率から、当てずっぽう成分をちょうど差し引いた真の既知率」の推定値であり、擬似語への誤反応率 f を「その受験者の当てずっぽう傾向」の直接推定量として使っている。

### 1.3 数値例

ある1k帯で、実在語5問・擬似語3問（計8問）を出題したとする。

| | 出題数 | Yes と答えた数 | 率 |
|---|---|---|---|
| 実在語 | 5 | 4 | h = 4/5 = 0.80 |
| 擬似語 | 3 | 1 | f = 1/3 = 0.333 |

```
p = (0.80 − 0.333) / (1 − 0.333) = 0.467 / 0.667 ≈ 0.70
```

補正後スコアは約70%——素点(h=0.80)よりも厳しく出る。これは「擬似語にも1/3の確率でYesと言う当てずっぽう傾向がある人」の実在語Yes率は、本当に知っている分だけでなく当てずっぽう分も混ざっている、という前提を反映した結果。

**縮退ケース**: f が高すぎる（例: 3問中2問以上が擬似語Yes）と p は信頼できない（当てずっぽうの寄与が大きすぎる、または f=1 で分母ゼロ）。この場合は補正計算そのものより先に「このプローブは信頼できない」と判定し、再プローブする（§3.3）。

## 2. アイテムバンク設計

### 2.1 実在語（データ生成不要）

- **供給源**: そのコースの `getCards()` が返す配列を `frequencyRank` から帯インデックス（`Math.floor(frequencyRank / 1000)`）でグルーピングするだけ。新規データ生成は不要。
- **サンプリング**: 1プローブあたり5語。帯内のカード集合から、以下を除外してからランダム抽出:
  - `gloss` が空/プレースホルダーのカード（生成漏れ）
  - `pos` が機能語（`particle` / `conjunction` / `interjection` 等）——意味有無判定・4択の題材として不向き。10k帯以上では実質的にこの除外の影響は小さい（機能語はほぼ必ずbaseline側=10k未満に来るため）
  - 同一セッション内で既出のカード（reprobe時の重複防止）
- **帯あたりの必要語数**: 5語/プローブ × 最大2プローブ（初回＋reprobe）= 最大10語。各帯とも実データは数百〜千語規模あるため枯渇の心配はない。

### 2.2 擬似語（新規生成が必要）

擬似語は**実在語と同じ驚きやすさ（surface plausibility）**を持たせる必要がある。特に日本語では「漢字複合語は字面から意味推測できてしまう」問題（PLAN §3.2）があるため、擬似語もその推測誤りを模倣する形で作る——そうしないと擬似語だけ不自然に「見るからに偽物」になり、f が実際の過大申告リスクを過小評価してしまう。

#### 日本語（コースE）

- **主方式（約80%）: 漢字再結合**。対象帯（±3帯）の実在見出し語で使われている漢字プールから、2文字（帯内の見出し語の主流である2字熟語の比率に合わせる。3字も一定数混ぜる）をランダムに組み合わせて新しい熟語を生成する。
  - **非実在チェック**: 生成した熟語が `pipeline/raw/jmdict-eng-3.6.2.json`（23.5万エントリ・既にダウンロード済み）の見出し語（漢字表記・かな表記の両方）に一致しないことを確認。一致すれば棄却して再生成。
  - **読みの割当**: 新規のKANJIDIC2ダウンロードを増やさず、**JMdict内で当該漢字を含む既存の複合語から観測された音読みの断片を再利用**して読みを組み立てる（同じ漢字が実在の熟語でどう読まれているかを流用するので、読みも「もっともらしい」ものになる）。
  - この方式は「字面から意味を推測できてしまう」という実在語の弱点を擬似語にも意図的に再現するための設計選択——これにより f が本当に測りたい「漢字部品からの当てずっぽう傾向」を捕捉できる。
- **副方式（約20%）: かな文字列**。和語・外来語らしい仮名の並び（モーラ長を帯内の実在語の長さ分布に合わせる）をランダム生成し、同様にJMdict非存在チェックを通す。漢字の透けた推測に頼らない当てずっぽう傾向も別途捕捉するため。

#### 英語（コースA）

- **方式: 文字トライグラムMarkov連鎖**。対象帯の実在見出し語（10k〜30k帯はラテン語源の複雑な語が主流）でトライグラムモデルを学習し、実在語の長さ分布に合わせた新規文字列を生成する。学術的に確立された同種手法として Wuggy（Keuleers & Brysbaert 2010、非単語生成の標準ツール）があり、本パイプラインはその簡易版（追加ライブラリ非依存の自前トライグラムモデル）に相当する。
  - **非実在チェック**: 大規模英語ワードリストと突合し、実在語との衝突を除外する。**未着手の依存関係**: 現状 `pipeline/raw/` に英語ワードリストが無い（Tatoeba・JMdict・JLPTリストのみ）。Phase 5 で SCOWL 等のオープンな英語ワードリストを新規取得する必要がある（open question §7）。

### 2.3 アイテムバンクJSON形状

擬似語のみビルド時に生成し、小さな静的JSONとして配布する（20帯 × 6語 ≈ 120エントリ、数KB程度——容量面の懸念は無い）。

```json
// public/data/placement/<courseId>/pseudowords.json
{
  "courseId": "ja-10-30k",
  "generatedAt": "2026-08-01",
  "method": "kanji-recombine(80%) + kana-string(20%), filtered against JMdict headwords",
  "bands": [
    {
      "band": 10,
      "pseudowords": [
        { "id": "ja-10-30k-pw-010-0", "headword": "塑徴", "reading": "そちょう", "method": "kanji-recombine" },
        { "id": "ja-10-30k-pw-010-1", "headword": "錬拡", "reading": "れんかく", "method": "kanji-recombine" }
        // ... 帯あたり6件（初回プローブ3件＋reprobe用の予備3件）
      ]
    }
    // ... band 11 〜 29（計20帯）
  ]
}
```

英語版も同形状（`reading` フィールドは省略可、`method: "trigram-markov"`）。

**帯あたり6件保持する理由**: 1プローブで擬似語3件を使う。信頼性ガード（§3.3）で同一帯を re-probe する場合、既出3件と重複しない新しい3件が必要なため、合計6件をあらかじめ用意しておく。

## 3. 第1段：Yes/Noテストのスコアリング＆既知フロンティア決定ロジック

### 3.1 パラメータ

| 定数 | 値 | 意味 |
|---|---|---|
| `BAND_MIN` / `BAND_MAX` | 10 / 29 | コースA・Eの帯インデックス範囲（10k〜30k） |
| `PROBE_SIZE` | 8 | 1プローブ = 実在語5 + 擬似語3 |
| `KNOWN_THRESHOLD` | 0.70 | 補正スコア p がこれ以上で「既知帯」判定 |
| `UNRELIABLE_F` | 0.67 | 擬似語のfalse-alarm率がこれ以上（3問中2問以上Yes）で信頼性ガード発火 |
| `SCAN_STEP` | 4 | 粗いスキャン段階の帯の飛び幅 |
| `GLOBAL_LOWCONF_MEAN_F` | 0.50 | セッション全体の平均fがこれ以上でセッション全体を lowConfidence 扱い |

各プローブは実在語5件・擬似語3件を**混ぜてシャッフル**して提示する（受験者が並び順から実在/擬似を見抜けないようにする）。

### 3.2 アルゴリズム（seed → scan → bisect）

1. **Seed（帯10で健全性チェック）**: コースの開始帯（学習者全員が既知と仮定しているbaseline）そのものを1プローブ実施。
   - p < `KNOWN_THRESHOLD` → **即終了**。フロンティア = 帯10未満（＝skip対象なし）。「baseline未確認」フラグを立てて通常の学習開始へ（帯10自体が既知でない学習者は、コース設計上のbaseline前提が崩れているケース——UIでは静かに通常オンボーディングへ流す）。
   - p ≥ `KNOWN_THRESHOLD` → 2へ。
2. **粗いフォワードスキャン**: 帯 10+4, 10+8, 10+12, … と `SCAN_STEP=4` 刻みで帯29まで（最大5プローブ）、p ≥ `KNOWN_THRESHOLD` の間だけ前進を続ける。
   - 最後まで全て既知判定だった場合 → フロンティア = 帯29超（コース範囲を使い切った稀なケース。open question §7）。
   - 途中で初めて p < `KNOWN_THRESHOLD` の帯が出たら、そこを `bandFail`、直前の既知帯を `bandPass` として3へ（この時点で両者の差は最大4帯）。
3. **二分探索での精緻化**: `bandPass` と `bandFail` が隣接するまで、中点の帯を繰り返しプローブ（差4なら最大2プローブで隣接に収束）。
   - 中点が既知 → `bandPass` を更新。未知 → `bandFail` を更新。
4. **粗いフロンティア確定**: `bandPass` が「Stage 1が既知と判定した最後の帯」= `coarseFrontierBand`。フロンティアランク（暫定）= `(coarseFrontierBand + 1) × 1000`。

**想定アイテム消費**: 最悪ケースで seed(8) + scan(最大5×8=40) + bisect(最大2×8=16) = 64問。典型ケースは40〜56問程度。PLAN §3.2 が見積もる「計60-120語」のレンジ内〜やや下限寄りで収まる（余裕を持たせたい場合は `PROBE_SIZE` を10（実在6+擬似4）に増やすと典型60〜80問に調整可能——チューニングパラメータとして残す）。

### 3.3 信頼性ガード

- **プローブ単位**: あるプローブの f ≥ `UNRELIABLE_F` の場合、そのプローブの結果を採用せず、同じ帯を**未使用の擬似語3件＋新しい実在語5件**で1回だけ re-probe する。
  - re-probe後も f ≥ `UNRELIABLE_F` → そのプローブ結果は「信頼できない」まま確定させ、**保守的に unknown 判定**する（実在語のh値に関わらず。既知への誤爆＝過剰スキップの方が、未知への誤爆＝1回多くレビューさせるより実害が大きいため、疑わしきは「未知」側に倒す）。セッションに `lowConfidence: true` を立てる。
- **セッション単位**: 全プローブのfの平均が `GLOBAL_LOWCONF_MEAN_F` 以上なら、個々のプローブ結果に関わらずセッション全体を `lowConfidence: true` とし、UIでは「回答が一貫していなかったため保守的に配置しました。いつでも再受験できます」といった案内を出す（文言はPhase 4のUI検討事項）。
- p が負値になる場合（f > h）は 0 にクランプする。f = 1（分母ゼロ）は `UNRELIABLE_F` 判定で事前に弾かれるため p の計算自体が発生しない。

## 4. 第2段：4択意味マッチによる境界調整

Stage 1 のYes/No＋擬似語補正だけでは、**日本語の漢字複合語の「字面から意味推測できてしまう」問題を完全には補正できない**（これはPLAN §3.2 が名指しで警告している既知のリスク）。擬似語側も同じ推測しやすさを再現するよう作ってはいるが（§2.2）、実在の未知語自身が「知らないのにYesと言いたくなる」誘因を持つ以上、真に意味を分かっているかを直接確認する第2段が必要——これが「意味マッチ比重を上げる」の実装。

### 4.1 サンプリングと採点

- **アイテム**: Stage 1 と異なり新規データ生成は不要——対象語（帯 `coarseFrontierBand` 付近）の実在カードに対し、**同じ帯±3帯内・同じ `pos` の他カードのglossを誤答選択肢として実行時にランダム抽出**し、正解glossとシャッフルして4択を作る（事前生成の別ファイルは持たない。カードデータは既にメモリ上にあるため純粋関数で完結）。
- **プライマリプローブ**: `coarseFrontierBand`（Stage 1が「既知」と判定した境界帯）から10問。
- **チャンス補正式**（k択の当てずっぽう補正——Stage 1の高閾値モデル補正と同じ「補正for guessing」の系譜に属する、教育測定で標準的な式）:
  ```
  corrected = (accuracy − 1/k) / (1 − 1/k)
  ```
  k=4 のとき `corrected = (accuracy − 0.25) / 0.75`。

### 4.2 分岐ロジック（境界±1帯調整）

| プライマリ結果 | 分岐 | 追加プローブ | 調整 |
|---|---|---|---|
| `corrected ≥ 0.70` | **Branch A（確認済み）** | `coarseFrontierBand + 1`（Stage 1の未知帯）から5問 | 追加プローブも `corrected ≥ 0.70` なら **+1帯**（フロンティアを1帯上げる）。そうでなければ調整なし（+0） |
| `corrected < 0.70` | **Branch B（過大申告を検出）** | `coarseFrontierBand − 1` から5問 | 追加プローブが `corrected ≥ 0.70` なら **−1帯**（フロンティアを1帯下げる＝境界帯自体を未知へ再分類）。そうでなければさらに1帯下げて安全側に倒す（open question §7: この場合カスケードを続けるか、−1で打ち切るか） |

**アイテム消費**: 10（プライマリ）+5（分岐）= 15問。PLAN §3.2 の「10-20問」レンジ内。

**最終フロンティアランク** = `(最終確定帯 + 1) × 1000`。例: 最終確定帯=15 → フロンティアランク=16000（ランク1〜16000未満を既知baselineとして一括処理、16000以降は通常のFSRS学習キューへ）。

## 5. エンドツーエンド数値例

コースE（ja-10-30k）、架空の受験者での通し例。

1. **Seed（帯10）**: 実在5/5 Yes、擬似1/3 Yes → h=1.0, f=0.333 → p=(1.0−0.333)/0.667=1.0 → 既知。続行。
2. **スキャン 帯14**: h=4/5=0.8, f=1/3=0.333 → p=(0.8−0.333)/0.667=0.70 → 既知（境界値）。続行。
3. **スキャン 帯18**: h=3/5=0.6, f=2/3=0.667 → f≥UNRELIABLE_F(0.67は僅かに届かないので通常計算) p=(0.6−0.667)/0.333=−0.20 → 0にクランプ → 未知。`bandPass=14`, `bandFail=18`。
4. **二分探索 帯16**: h=4/5=0.8, f=0/3=0 → p=0.8 → 既知。`bandPass=16`。
5. **二分探索 帯17**: h=3/5=0.6, f=1/3=0.333 → p=(0.6−0.333)/0.667=0.40 → 未知。`bandFail=17`。`bandPass`と`bandFail`が隣接 → 打ち切り。
   → **粗いフロンティア = 帯16**（暫定ランク17000未満）。
   Stage 1 消費: seed(8)+帯14(8)+帯18(8)+帯16(8)+帯17(8) = 40問。
6. **Stage 2 プライマリ（帯16、10問）**: 正答6/10 → accuracy=0.6 → corrected=(0.6−0.25)/0.75=0.467 <0.70 → **Branch B（過大申告検出）**。
7. **Stage 2 分岐（帯15、5問）**: 正答5/5 → accuracy=1.0 → corrected=1.0 ≥0.70 → **−1帯調整**。
   Stage 2 消費: 10+5=15問。
   → **最終フロンティア = 帯15**（Stage 1が「既知」と誤判定した帯16は取り消される）。

**最終フロンティアランク = 16000**。合計消費アイテム数 = 40+15 = 55問（PLAN見積もり60-120語のレンジにほぼ収まる／やや下振れ、§3.2末尾のチューニング余地で調整可）。ランク10000〜15999は既知baselineとして一括処理、16000以降が通常学習キューの起点になる。

## 6. データモデル（TypeScript interface スケッチ）

既存パターン（`courseId` でスコープ、`WordProgress`/`CourseSummary` の設計）に合わせる。

```typescript
// --- 配置セッション全体 ---
/** 配置テストのセッション状態。コースA・Eのみ想定（'en-10-30k' | 'ja-10-30k'）。 */
export interface PlacementSession {
  courseId: CourseId
  startedAt: string // ISO
  status: 'stage1' | 'stage2' | 'done' | 'abandoned'
  stage1: PlacementStage1State
  stage2?: PlacementStage2State // stage1完了後に生成
  result?: PlacementResult // status='done' で確定
}

// --- Stage 1: Yes/No ---
export interface PlacementYesNoItem {
  itemId: string
  type: 'real' | 'pseudo'
  cardId?: string // type='real' のときのみ。VocabCard.id への参照
  headword: string
  reading?: string
  response?: 'yes' | 'no' // 未回答は undefined
}

export interface PlacementProbe {
  band: number // 10..29
  items: PlacementYesNoItem[] // 実在5+擬似3（reprobe時も同形）
  hitRate: number // h
  falseAlarmRate: number // f
  correctedScore: number // p = clamp((h-f)/(1-f), 0, 1)
  verdict: 'known' | 'unknown'
  unreliable: boolean // f >= UNRELIABLE_F だったか（reprobe後も解消しなかった場合 true のまま確定）
  isReprobe: boolean
}

export interface PlacementStage1State {
  probes: PlacementProbe[] // 実施順に追記（seed→scan→bisect）
  phase: 'seed' | 'scan' | 'bisect' | 'complete'
  bandPass?: number
  bandFail?: number
  coarseFrontierBand?: number // phase='complete' で確定
  baselineUnconfirmed: boolean // seedで帯10落選 → 即終了したケース
}

// --- Stage 2: 4択意味マッチ ---
export interface PlacementChoiceItem {
  itemId: string
  cardId: string
  headword: string
  correctGloss: string
  choices: string[] // 4件（correctGlossを含みシャッフル済み）
  response?: string // 未回答は undefined
}

export interface PlacementStage2Probe {
  band: number
  items: PlacementChoiceItem[] // 10件（primary）or 5件（adjust）
  accuracy: number
  correctedScore: number // (accuracy - 0.25) / 0.75
  verdict: 'known' | 'unknown'
}

export interface PlacementStage2State {
  primary: PlacementStage2Probe // coarseFrontierBand に対する10問
  adjustProbe?: PlacementStage2Probe // Branch A/B で発生する追加5問
  adjustment: -1 | 0 | 1 // 最終的な帯の増減
}

// --- 結果 ---
export interface PlacementResult {
  frontierRank: number // 例: 16000。この値未満のランクを既知baselineとして一括処理
  finalBand: number // frontierRank の元になった帯インデックス
  lowConfidence: boolean // 個別プローブ or セッション全体のfから判定
  completedAt: string
}
```

**適用（progress書き込み）の設計方針**: `PlacementResult.frontierRank` が確定したら、`frequencyRank < frontierRank` の全カードに対し、**既存の `recordReview(card, 'good')` を1回ずつ適用したのと等価な状態**にする（新しい採点ロジックを発明しない。プレースメントの「既知」判定は「learnerが一度 “I know” を押した」のと同じ意味論として扱う）。ただしフロンティア対象は最大 6,000〜16,000語規模になり得るため、`recordReview` を1件ずつ呼ぶ（毎回トランザクション＋summary/dailyStats再計算）実装はそのままでは性能上厳しい——**1トランザクションで完結するバルク版**（`applyPlacementResult(courseId, frontierRank)` 相当。summaryは全体を1回再集計、dailyStatsは`newStarted`をまとめて加算）が必要になる（open question §7）。'burned' ではなく通常の 'known' として書き込む（stability は1回のgood評価分のみ＝将来のFSRS復習で自然に再検証され、誤ったフロンティア判定はその時点で修正される安全弁になる）。

## 7. 未決事項（実装時に判断が必要）

1. **`recordReview` のバルク版**: 最大16,000件規模の一括「既知」書き込みを1トランザクションで行う `applyPlacementResult` を新設するか、既存 `recordReview` をループで呼んで実測してから決めるか。性能実測が先。
2. **英語ワードリストの未取得**: 擬似語の非実在チェックに使う大規模英語ワードリスト（SCOWL等）が `pipeline/raw/` に無い。Phase 5 のデータ調達タスクとして計上が必要。
3. **カスケードの打ち切り**: Stage 2 Branch B で `coarseFrontierBand − 1` も失敗した場合、さらに `− 2` を試すか、そこで安全側に倒して打ち切るか（本書は「打ち切り」を既定としたが、アイテム予算（10-20問）を守るなら打ち切り一択、緩めるなら再検討）。
4. **帯29を超えて全帯既知だった場合の扱い**（Stage 1 フォワードスキャンが最後まで既知判定）: 「このコースはもう卒業レベル」というレアケース。単純に帯29までのフロンティアで打ち切って通常学習へ流すか、Phase 6のマイニング機能への誘導メッセージを出すか。
5. **再受験（retake）時のデータ方針**: `PlacementSession` を `courseId` 主キーで1行のみ保持し前回を上書きするか、履歴として複数保持するか（本書は「上書き・履歴なし」を推奨——既存 `summary` テーブルの設計思想（派生データは常に1系統）に合わせる軽量な選択）。
6. **出題の時間制限の要否**: Yes/Noテストは伝統的には即答性を重視する（辞書を引く暇を与えない）。時間圧をUIで課すか、性善説で無制限にするかはUX判断（本書のスコアリングロジック自体には影響しない）。
7. **`PROBE_SIZE` の最終値**: 本書は5実在+3擬似=8を既定としたが、§3.2の消費数試算はPLANの「60-120語」レンジのやや下限寄り。実際のユーザーテストで体感時間（3-5分目標）を測ってから、8→10（6実在+4擬似）への引き上げを検討してよい。
8. **Stage 2 の誤答選択肢の質**: 「同じ帯±3帯・同じpos」というヒューリスティックのみで真の同義語が誤答に紛れ込むリスクをどこまで許容するか。埋め込みベースの意味類似度フィルタ導入は本書ではオーバーエンジニアリングと判断し見送ったが、実データで誤答の質が悪いと分かれば再検討。
